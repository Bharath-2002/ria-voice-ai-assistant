# Ria — Write‑up: decisions, trade‑offs, edge cases

## Architecture decisions

**ElevenLabs Agents platform for the conversation, FastAPI only for tools.**
The hard parts of a voice agent — barge‑in/turn‑taking, ASR, TTS, the audio WebSocket, an LLM loop with tool calling, the post‑call analysis — are exactly what the ElevenLabs platform does well. Building that on raw Twilio media streams would be weeks of work for a worse result. So the agent (voice, system prompt, LLM, `end_call`, dynamic variables, the 5 tools) lives in ElevenLabs, and our backend is just a thin, stateless‑ish HTTP service that the agent calls: hit BlueStone, shape a voice‑friendly response, send WhatsApp, find a store, place an outbound call. Twilio is connected to the agent via ElevenLabs' "import phone number", so inbound audio streams straight to the agent — our backend isn't in the audio path at all.

**Layered backend with constructor DI.** `api → features → services → repositories → entities/shared`, wired in one container (`app/api/container.py`). It's a small app, but the seams make it easy to test a service in isolation and to swap implementations (e.g. Redis → Postgres for sessions later) without touching the routes. Failures are contained at the boundary: a tool route never returns a 500 to ElevenLabs if it can help it — it returns a `{"say": "...graceful sentence...", "data": {...}}` so Ria says something sensible instead of going silent.

**State: Redis for the live call, Postgres for durable memory + evals, ElevenLabs for the record.**
Each tool call from ElevenLabs is an independent stateless HTTP request — there is no in‑process session, and the backend can run as multiple replicas. So anything shared *between* tool calls of one conversation needs an external store. That's **Redis**: keyed by the ElevenLabs conversation id (every tool carries it via the `system__conversation_id` dynamic variable), TTL‑expiring, holding the customer's captured preferences, the full catalogue results of the current search (so `send_to_whatsapp` can build cards and page through "a few more" while the agent passes only IDs), and the running list of products actually sent to WhatsApp. **Postgres** holds the durable, cross‑call memory — `customers` and `conversations` — plus evaluation results. The authoritative call record (transcript, tool calls, latency, ElevenLabs' own summary) is owned by **ElevenLabs**; mirroring it would be a cache with no upside, so the eval dashboard reads it on demand. (The LLM's turn‑to‑turn conversational memory — "the second one", "something cheaper" — is ElevenLabs‑side; Redis is the *backend's* shared working store, not a re‑implementation of that.)

**Cross‑call memory.** When a call ends, `/elevenlabs/post-call` spawns a background task: `MemoryService.summarize_and_save` pulls the Redis session (preferences + recommended products + the rich `sent_products_full` list of what actually went to WhatsApp), calls Gemini (`gemini-2.5-flash`, JSON mode, temp 0) for a structured `{summary, outcome, follow_up, customer_name}`, upserts the customer (find‑or‑create by normalised phone; `COALESCE`s the name if newly captured), and upserts the conversation row (UNIQUE on the ElevenLabs conversation id — idempotent for webhook retries). On the next call, both `/elevenlabs/initiation` (inbound) and `/voice/outbound` (outbound) look up the last 3 conversations and inject them into the agent as the `previous_conversations` dynamic variable — header + per‑call line (relative time + outcome + the LLM summary) **plus** the design IDs of the products that actually landed on the customer's WhatsApp during that call. With those IDs in hand, Ria can act directly: *"resend the Tatva mangalsutra"* becomes `send_to_whatsapp(design_ids=[71785])`, no catalogue search.

**`send_to_whatsapp` as its own tool, not a side‑effect.** Earlier the search/details tools auto‑sent cards. Making `send_to_whatsapp` an explicit, on‑request tool is better UX *and* better data — Ria asks "interested in any of these? I'll WhatsApp the link", then sends; can page through the rest ("a few more designs?"); and every send updates `sent_products_full` in the session so the post‑call summariser knows exactly what the customer has on their phone, not just what was talked about.

**Eval framework: hybrid judge, not LLM‑only.** Subjective things (did she follow the discovery flow? was the tone natural? is the post‑call summary accurate?) are graded by a Gemini judge with a strict rubric, JSON mode, temperature 0. Things that must never happen and are cheap to check deterministically (raw JSON/URLs/IDs spoken, truncated turns, double greeting, latency, "was `send_to_whatsapp` actually called when the customer agreed") are checked in Python and never trusted to the LLM. Scoring gates on **per‑dimension thresholds + critical checks** rather than one blended number, so a great conversation that leaked raw JSON still fails.

**uv + Alembic for the deployment surface.** `uv` is the only package manager (lockfile + reproducible `uv sync --frozen` in the Docker image, no pip/poetry split). Alembic owns the Postgres schema for `customers` and `conversations`: `alembic upgrade head` runs in the container start command before uvicorn, so every deploy lands the schema before the app expects it. The Alembic `env.py` uses an `include_object` callback so it never touches the eval framework's `evaluations` table (which lives in the same Postgres but is owned by `eval/store.py` and created via `create_all`). Schema changes become *"edit the model, generate a revision, deploy"* — no manual SQL, no drift between dev and prod.

**Why Streamlit for the eval dashboard.** Fastest way to a real, clickable UI in pure Python — list view with filters and multi‑select, a detail view with the transcript and per‑check reasoning, and buttons that trigger validation. Deployed as a second Railway service from the same repo (`Dockerfile.dashboard` selected via `RAILWAY_DOCKERFILE_PATH`), sharing the project's Postgres.

## Trade‑offs between models and design

- **TTS model:** Eleven Flash v2.5 (~75 ms) over Turbo v2.5 — for a phone call, latency beats the marginal quality bump.
- **Agent LLM:** a "flash/mini/haiku"‑tier model is the right call — Ria does structured discovery + reliable tool calling, not deep reasoning, so the latency/cost of a frontier model isn't worth it. Gemini‑Flash‑class is the sweet spot; GPT‑4o‑mini is the safe fallback if tool‑calling reliability ever wobbles.
- **Post‑call summariser + eval judge:** both use `gemini-2.5-flash`. Cheap, fast, good at structured JSON with temp 0. Trade‑off for the summariser is LLM variance on a single shot — mitigated by giving it the structured in‑session data (preferences, recommended, sent_products) explicitly so the prose has facts to lean on; trade‑off for the judge is the same, and could be reduced later by sampling 2–3× and taking the majority.
- **Two‑table customer/conversation schema (vs single denormalised table):** since we now capture a customer's *name* (and will likely add CRM‑style attributes), modelling `customers` and `conversations` separately from day one avoids a later migration. `cards_sent` is JSONB (rich dicts, not just IDs), so the schema accommodates richer content without ALTERs.
- **Slim vs full tool responses:** returning all 10 full product dicts to the agent is unnecessary context; returning only `id/name/price` for the list (full dicts stay in Redis for the WhatsApp cards) keeps the LLM context tight.
- **Budget as a range tag, no client‑side filtering:** the BlueStone API *does* filter by `rs <from> to <to>` correctly — *if it's the only tag*. So we fold metal/occasion/stone into the query text and send only the budget tag, and trust the API. No client‑side price filtering — it'd be a band‑aid over a working API once you use it right.
- **No Postgres mirror of full ElevenLabs calls:** keeps the system simpler (no sync job, no staleness). Cost: the dashboard makes API calls to ElevenLabs on each refresh. Fine at this scale; you'd add a mirror only if you needed offline access or queries over tens of thousands of calls.
- **Batch validation on a small thread pool (4):** parallel enough to be useful, bounded so it stays under the Gemini RPM limit and the DB pool. `validate()` is sync I/O, so threads (not asyncio) without a rewrite.
- **BlueStone proxy hook, lazy by default:** `BLUESTONE_PROXY` env var. When set, `BlueStoneService` builds its own httpx client routed through the proxy (a Webshare rotating endpoint in production); when unset, the existing shared client is reused. So the same code path serves "no proxy" dev and "proxy in front of cloud IP" prod without conditional branches in the call sites.

## Edge cases handled

- **BlueStone 403s from cloud IPs** — the catalogue intermittently blocks requests from datacenter egress IPs. Three layers of mitigation: (1) browser‑like headers (User‑Agent, Accept‑Language, Referer), (2) a 3‑attempt exponential‑backoff retry on connection errors and 403/429/5xx, (3) the `BLUESTONE_PROXY` env‑var hook routes calls through a residential rotating proxy in production. If everything fails, Ria says "I ran into a problem searching the catalogue — could you repeat your preferences?" instead of crashing.
- **Empty search results** — Ria offers alternatives (broaden the budget, different metal/style) rather than dead‑ending.
- **Inbound vs outbound caller number** — on inbound, `system__caller_id` is the customer; on outbound it's the *agent* line, so `/voice/outbound` injects the dialed number as the `outbound_customer_phone` dynamic variable. The system prompt has a blunt "read both values explicitly" rule. Either way, Ria confirms the number before the first WhatsApp send and uses a different one if the customer asks.
- **Phone normalisation** — `app/services/phone.py` is the single source of truth. `whatsapp:+91…`, raw 10‑digit, leading 0 — all map to one canonical `+<E.164>` so the same person never forks into two `customers` rows.
- **Store lookup by place name** — a customer can say "Koramangala" instead of a pincode; `StoreService` resolves via `api.postalpincode.in`, then queries the BlueStone store locator; "no store nearby" → Ria offers the online links instead. When the customer asks to text the addresses, **all** nearby stores get sent (one WhatsApp message per store), not just the closest one.
- **Double greeting / call running on** — the system prompt has a hard "greet only once" rule (if the customer says "hi" back, acknowledge briefly and move to discovery, don't re‑introduce), and Ria uses the `end_call` tool to wrap up. The eval framework's deterministic `no_double_greeting` and `call_ended_cleanly` checks catch regressions.
- **One question at a time** — promoted to a "CRITICAL RULE" in the prompt with wrong/right examples; the original Step 4 (which itself bundled metal + stone) was split. Enforced as a *critical* check in the eval rubric.
- **Tool latency / dead air** — every tool has `pre_tool_speech: force`, and the prompt gives Ria filler lines ("let me pull that up for you…") so there's no silence while a tool runs.
- **Idempotent persistence** — `conversations.conversation_id` is UNIQUE and writes go through `INSERT … ON CONFLICT DO UPDATE`, so retried post‑call webhooks are no‑ops. `customers` upserts COALESCE the name so a later call's captured name doesn't overwrite an earlier capture.
- **Memory falls back gracefully** — if `DATABASE_URL` or `GEMINI_API_KEY` is missing, `MemoryService` simply isn't instantiated; the post‑call webhook skips the summary task, `previous_conversations` is empty, and Ria treats every caller as new. Local dev without Postgres still works.
- **Webhook idempotency / signature** — post‑call webhook HMAC‑verifies when `ELEVENLABS_WEBHOOK_SECRET` is set (with a 5‑minute timestamp window), accepts otherwise. Earlier issues where ElevenLabs auto‑disabled the webhook after repeated 403s are mitigated by `retry_enabled: true` on the webhook config.

## What I'd do with more time

- **Voice‑quality eval from the audio**, not the transcript shape — right now audio quality is judged manually; I'd run the ElevenLabs recording through some automated analysis (VAD/overlap, ASR confidence, dead‑air detection) so the Voice Quality dimension stops relying on transcript shape alone.
- **Scheduled call‑backs** — `conversations.follow_up` is already captured as free text; the next iteration parses it to a timestamp, queues it in a new `scheduled_callbacks` table, and a worker fires `/voice/outbound` at that time with the customer's prior‑call context already in scope.
- **Approved WhatsApp Content Templates + a real WhatsApp Business sender** so cards deliver outside the 24‑hour window (the code path already exists behind `WHATSAPP_TEMPLATE_SID`).
- **Nightly eval job** — auto‑validate the day's calls and post a summary (pass rate, critical failures) to Slack; the dashboard is on‑demand only today.
- **Prompt builder loop** — feed the recurring issues surfaced by the eval framework into a job that proposes (or applies, with review) prompt updates automatically. Closes the loop between eval and the thing being evaluated.
- **Indian phone number for production** — currently a US Twilio number; a production deployment under an Indian business entity would use a Twilio India regulatory bundle, or an Indian CPaaS (Plivo/Exotel) bridged into ElevenLabs over SIP.
- **Tests** — unit tests for the services (especially `BlueStoneService` query composition, the budget‑tag logic, `MemoryService` formatting, phone normalisation) and a contract test against the BlueStone API shapes.
